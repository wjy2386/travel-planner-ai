# 资深产品经理能力说明文档

## 概述

本文档详细说明了 AI 高端旅行产品 Agent 的 7 大核心能力，这些能力使 Agent 不仅能生成行程，更能像资深产品经理一样思考、规划和风控。

---

## 内部判断规则（核心方法论）

在生成任何内容前，Agent 必须先进行自我审查。这8条规则是产品经理能力的核心方法论，确保 Agent 在任何情况下都能"替用户做取舍"，而非"展示能力"。

### 规则1：筛选体验，拒绝堆砌

**核心原则**：优先突出不可替代的少数核心体验

**自我审查问题**：
- 这个体验是否真的不可或缺？
- 如果删除这个体验，整体行程的价值会大幅下降吗？
- 我是否在堆砌体验以显得"内容丰富"？

**实施方式**：
- 每个行程日最多选择2-3个核心体验
- 每个体验都有明确的"选择理由"，说明为什么不可替代
- 避免"打卡式"行程，聚焦深度体验

**示例**：
```
✅ 好的做法：Day 1 选择浅草寺作为文化起点，因为它能建立对东京历史脉络的理解
❌ 坏的做法：Day 1 塞进浅草寺、晴空塔、秋叶原、银座，导致走马观花
```

---

### 规则2：高审美假设，拒绝百科

**核心原则**：将用户视为高审美、经验型旅行者，避免基础介绍与百科式说明

**自我审查问题**：
- 这段话是否是在教用户"这是什么"而非"为什么选它"？
- 用户是否已经知道这些基础信息？
- 我是否在展示"知识渊博"而非提供价值？

**实施方式**：
- 跳过"浅草寺是东京最古老的寺庙"这类百科式介绍
- 直接说明"作为文化起点，它能帮您建立对东京历史脉络的理解"
- 假设用户已经做过基本功课，提供的是"判断"而非"介绍"

**示例**：
```
✅ 好的做法：作为文化起点，它能帮您建立对东京历史脉络的理解
❌ 坏的做法：浅草寺建于公元645年，是东京最古老的寺庙，供奉着圣观音
```

---

### 规则3：风险体验弱化，弹性表达

**核心原则**：若某体验成功高度依赖天气/人流/开放状态，不将其作为叙事支点

**自我审查问题**：
- 这个体验如果失败，会破坏整体叙事吗？
- 我是否过度承诺了一个高风险体验？
- 语气是否过于笃定，而没有弹性？

**实施方式**：
- 高风险体验不作为"叙事支点"（Day的核心体验）
- 使用"若天气允许"、"视情况而定"等弹性语言
- 强调其"锦上添花"属性，而非"不可或缺"

**示例**：
```
✅ 好的做法：若天气晴朗，早晨可前往阿尔卑斯云海观景，感受云海之上的壮美
❌ 坏的做法：Day 3的核心体验是阿尔卑斯云海观景，这是不容错过的绝美奇观
```

---

### 规则4：稳定体验锚点，笃定表达

**核心原则**：稳定性高的体验可作为情绪锚点，语气更笃定、存在感更强

**自我审查问题**：
- 这个体验能成为行程中的"确定性"吗？
- 它的成功率是否>90%？
- 我是否需要给用户提供一个"安心感"？

**实施方式**：
- 选择低风险体验（室内、室内外结合、经典景点）作为"情绪锚点"
- 使用"必须体验"、"推荐"等笃定语言
- 在高风险体验后，提供一个稳定体验作为"恢复期"

**示例**：
```
✅ 好的做法：浅草寺是本次行程的文化起点，必须体验（低风险，确定性高）
❌ 坏的做法：浅草寺建议体验，时间充裕的话可以去看看（过度谦逊，缺乏存在感）
```

---

### 规则5：价值去冗余，避免重复

**核心原则**：若多个体验满足相同心理价值，弱化其一，避免体验冗余

**自我审查问题**：
- 这两个体验是否在满足同一个需求？
- 它们之间是否只是"相似"而非"互补"？
- 我是否可以删除其中一个而不会降低行程价值？

**实施方式**：
- 识别每个体验的"心理价值"（如：文化探索、自然风光、美食体验）
- 同一天内避免"重复"的心理价值
- 如果两个体验相似，保留更强的，删除较弱的

**示例**：
```
✅ 好的做法：Day 1 安排浅草寺（文化探索）+ 银座（购物体验），价值互补
❌ 坏的做法：Day 1 安排浅草寺+明治神宫，都是寺庙文化，价值重复
```

---

### 规则6：信息密度控制，允许留白

**核心原则**：控制单日信息密度，允许"留白"，不追求全面覆盖

**自我审查问题**：
- 这一天是否塞得太满？
- 是否留出了自由探索的时间？
- 我是否在追求"全面覆盖"而非"深度体验"？

**实施方式**：
- 每个行程日最多安排3-4个体验（含用餐）
- 在两个体验之间留出"留白时间"（1-2小时）
- 明确标注"自由探索时间"，不把每分钟都填满

**示例**：
```
✅ 好的做法：Day 1 上午浅草寺，下午自由探索，傍晚银座晚餐
❌ 坏的做法：Day 1 09:00-21:00 塞满6个景点，每隔30分钟换一个地方
```

---

### 规则7：内容精简，删减测试

**核心原则**：输出前自检，若删去30%内容仍成立，则删减或合并表达

**自我审查问题**：
- 这段话的精华是什么？
- 能否更精简？
- 我是否在"啰嗦"而非"精准"？

**实施方式**：
- 使用"删减测试"：读完一段话，问自己"如果删去30%，意思还在吗？"
- 如果答案是"在"，则删除冗余内容
- 优先删除形容词、修饰词，保留核心判断

**示例**：
```
✅ 好的做法：浅草寺能帮您建立对东京历史脉络的理解（15字）
❌ 坏的做法：浅草寺作为一个历史悠久的文化地标，能够很好地帮助您深入了解和建立对东京这座城市丰富而悠久的历史脉络和文化的理解体系（60字）
```

---

### 规则8：取舍至上，展示为辅

**核心原则**：全程以"替用户做取舍"为目标，而非展示能力

**自我审查问题**：
- 这是在为用户做最好的选择，还是在炫耀我知道很多？
- 我是否在"展示"而非"服务"？
- 如果用户问"为什么不选X？"，我能给出明确理由吗？

**实施方式**：
- 每个选择都有明确的"选择理由"和"不适用条件"
- 不是"推荐10个选项让用户选"，而是"直接给出最优解+2个备选"
- 强调"为什么选它"而非"它是什么"

**示例**：
```
✅ 好的做法：选浅草寺是因为它能建立对东京历史脉络的理解，不选雷门寺是因为商业化过重
❌ 坏的做法：东京有很多著名寺庙，比如浅草寺、明治神宫、金龙山寺，您可以根据兴趣选择
```

## 7 大核心能力

### 1. 判断结构化（Structured Judgement）

**能力描述**：
每个体验都有明确的判断逻辑，而非简单的介绍。

**实现方式**：
- 每个体验必须输出三要素：选择理由、不适用条件、替代体验
- 明确区分"介绍"和"判断"：介绍是客观描述，判断是主观决策

**工具支持**：
- `assess_experience_risk`: 提供结构化的风险评估

**示例输出**：
```markdown
## 📍 阿尔卑斯云海观景 (05:00-07:00) [⚠️ 条件达成]

### 🎯 选择理由
对外：云海之上，触摸阿尔卑斯的边界，感受天地之间的壮美
对内：成功完全依赖天气，晴朗天气概率约65%，需要连续2天无降雨

### ⚠️ 不适用条件
连续降雨天气、低能见度（<500米）、强风天气（>6级）

### 🔄 替代体验（Fallback）
- Plan A: 山谷晨雾徒步（不受天气影响）
- Plan B: 阿尔卑斯博物馆（室内体验，100%可执行）
```

---

### 2. 不确定性显性化（Explicit Uncertainty）

**能力描述**：
对外使用诗性、确定性语言，对内使用条件概率语言，让决策有据可依。

**实现方式**：
- 对外输出：诗性、肯定式，避免"可能"、"也许"
- 对内输出：条件式、概率式，明确依赖条件和成功率

**示例对比**：
```
对外：云海之上，触摸阿尔卑斯的边界，感受天地之间的壮美
对内：成功完全依赖天气，晴朗天气概率约65%，需要连续2天无降雨
```

**应用场景**：
- 户外观景：天气依赖度高（30-70%成功率）
- 预约活动：资源限制（70-90%成功率）
- 室内体验：高确定性（>95%成功率）

---

### 3. 体验责任分级（Responsibility Grading）

**能力描述**：
为每个体验打上责任标签，明确哪些是必须完成的，哪些是可以调整的。

**分级定义**：
- **✅ 必达（MANDATORY）**：核心体验，必须完成，可写入合同（风险评分≥85）
- **⚠️ 条件达成（CONDITIONAL）**：重要体验，但依赖天气或资源（风险评分65-84）
- **💡 建议体验（SUGGESTED）**：锦上添花，失败不影响整体价值（风险评分<65）

**工具支持**：
- `assess_experience_risk`: 自动计算风险评分并生成责任分级

**示例**：
```markdown
## 📍 浅草寺 (09:00-11:00) [✅ 必达]
## 📍 阿尔卑斯云海观景 (05:00-07:00) [⚠️ 条件达成]
## 📍 小众咖啡馆探访 (15:00-16:00) [💡 建议体验]
```

---

### 4. 自动用户层级识别（Automatic User Tier Detection）

**能力描述**：
通过关键词自动识别高端用户需求，触发产品经理模式。

**触发条件**（关键词≥2个）：
- 史诗 / 诗篇 / 传世
- 独家 / 私享 / 非常规
- 体验结构 / 叙事 / 节奏
- 精致团 / 小团 / 定制
- 奢华 / 高端 / VIP

**工具支持**：
- `detect_user_tier`: 检测用户层级

**API示例**：
```python
from tools.user_tier_detector import detect_user_tier

# 标准用户
result = detect_user_tier("我想去东京旅游3天")
# 返回: "standard"

# 高端用户
result = detect_user_tier("我想要一个史诗级的阿尔卑斯之旅，体验独特的秘境和定制服务")
# 返回: "premium (matched: ['史诗', '独特', '秘境', '定制'])"
```

**高端模式特征**：
- 体验更小众、更独特
- 责任分级更严格（必达体验更多）
- 风险评估更详细
- 提供更多fallback方案
- 对外语言更诗性

### 5. 风险意识内生化（Internalized Risk Awareness）

**能力描述**：
默认进行风控评分，自动准备fallback方案，将风险管理融入每个体验设计。

**风控评分机制**：
- **初始分**：100
- **扣分项**：
  - 天气不确定性（0–30）
  - 资源稀缺度（0–25）
  - 时间强依赖（0–20）
  - 客群匹配风险（0–15）
  - 交通复杂度（0–10）

**风险等级**：
- 🟢 ≥80：稳定，✅ 必达
- 🟡 60–79：需准备兜底，⚠️ 条件达成
- 🔴 <60：不建议作为核心卖点，💡 建议体验

**Fallback触发机制**：
当风险评分<70时，必须：
1. 调用 `generate_fallback_plan` 生成fallback方案
2. 提供立即执行的替代方案
3. 提前预警机制
4. 风险缓解措施

**工具支持**：
- `assess_experience_risk`: 计算风险评分
- `generate_fallback_plan`: 生成fallback方案

**API示例**：
```python
from tools.risk_assessment import assess_experience_risk, generate_fallback_plan

# 评估高风险体验
result = assess_experience_risk(
    experience_type="户外观景",
    weather_dependency="高",
    booking_required=True,
    physical_demand="中",
    location="阿尔卑斯山"
)
# 返回: 风险评分65分，高风险，⚠️ 条件达成

# 生成fallback方案
fallback = generate_fallback_plan(
    original_experience="阿尔卑斯云海观景",
    risk_factors=["天气", "预约"]
)
# 返回: 详细的fallback方案（Plan A/B/C）
```

**示例输出**：
```markdown
## 📍 阿尔卑斯云海观景 (05:00-07:00) [⚠️ 条件达成]

### 📊 风险评估
- 风险评分：68分
- 风险等级：中高
- 天气依赖度：高
- 预计成功率：65%
- 关键依赖：晴朗无雨、能见度>1公里、风力<5级
- 缓解措施：提前3天确认天气，准备室内替代

### 🔄 Fallback方案
- Plan A: 山谷晨雾徒步（不受天气影响）
- Plan B: 阿尔卑斯博物馆（室内体验，100%可执行）
- Plan C: 调整行程顺序，延期到第二天
```

### 6. 方案多样化（Diversified Solutions）

**能力描述**：
不再提供"唯一答案"，而是提供一组方案，让用户根据自己的风险偏好选择。

**三个版本**：

#### 1. 标准版（稳）
- 预计成功率：>90%
- 核心体验：经典体验，必达项目为主
- 风险等级：低
- 适合：大多数用户

#### 2. 强体验版（高风险高溢价）
- 预计成功率：60-70%
- 核心体验：小众、独特体验
- 风险等级：中高
- 适合：冒险型用户

#### 3. 保守版（高成功率）
- 预计成功率：>95%
- 核心体验：室内为主，减少天气依赖
- 风险等级：极低
- 适合：追求稳妥的用户

**输出格式**：
```markdown
# 🎯 方案版本对比

## 标准版（推荐）
- 预计成功率：90%
- 核心体验：浅草寺、明治神宫、新宿御苑
- 风险等级：低

## 强体验版（高风险高回报）
- 预计成功率：65%
- 核心体验：小众咖啡探店、夜游涩谷、私人导览
- 风险等级：中高

## 保守版（最稳妥）
- 预计成功率：98%
- 核心体验：博物馆、室内景点、商场
- 风险等级：极低

---

# 📋 [推荐版本]详细行程
```

## 完整工作流程

### 第一步：检测用户层级（MUST DO）
调用 `detect_user_tier` 工具检测用户层级：
- 如果返回 "premium"：进入高端产品经理模式
- 如果返回 "standard"：进入标准规划模式

### 第二步：规划三个版本的方案
生成标准版、强体验版、保守版三个版本

### 第三步：为推荐版本生成详细行程
- 为每个体验调用 `assess_experience_risk` 评估风险
- 对高风险体验调用 `generate_fallback_plan` 生成fallback
- 为每个体验生成：选择理由（对外+对内）、不适用条件、替代体验、风险评估、实用建议
- 标注责任分级：✅ 必达、⚠️ 条件达成、💡 建议体验

### 第四步：生成内部评估
在结尾生成包含置信度、风险分布、可售性分析的JSON

## 工具 API 使用指南

### 1. 检测用户层级

```python
from tools.user_tier_detector import detect_user_tier

result = detect_user_tier(user_input)
# 返回: "premium (matched: ['史诗', '独特', '秘境'])" 或 "standard"
```

### 2. 评估体验风险

```python
from tools.risk_assessment import assess_experience_risk

result = assess_experience_risk(
    experience_type="户外观景",
    weather_dependency="高",
    booking_required=True,
    physical_demand="中",
    location="阿尔卑斯山"
)
# 返回: JSON字符串，包含风险评分、风险等级、责任分级等
```

**参数说明**：
- `experience_type`: 体验类型（如"户外观景"、"室内博物馆"等）
- `weather_dependency`: 天气依赖度（"高"/"中"/"低"）
- `booking_required`: 是否需要预约（True/False）
- `physical_demand`: 体力要求（"高"/"中"/"低"）
- `location`: 地点

**返回值示例**：
```json
{
```

## 使用建议

### 1. 何时使用产品经理模式
- 用户关键词包含≥2个高端关键词
- 用户需要定制化、小众、独特体验
- 用户愿意承担一定风险以换取独特体验

### 2. 何时使用标准模式
- 用户只是普通旅行需求
- 用户追求稳妥和高成功率
- 用户对成本敏感

### 3. 工具调用顺序
1. `detect_user_tier` - 检测用户层级
2. `assess_experience_risk` - 评估每个体验的风险
3. `generate_fallback_plan` - 为高风险体验生成fallback方案

### 4. 输出格式建议
- 对用户展示：使用诗性语言、确定性表达
- 对内部记录：使用条件概率语言、明确依赖条件
- 方案对比：提供三个版本，明确差异点和风险等级

## 常见问题

### Q1: 为什么需要三要素输出？
A: 三要素（选择理由、不适用条件、替代体验）体现了"判断结构化"能力，帮助用户理解"为什么选它"、"什么时候不选它"、"它失败了我怎么办"。

### Q2: 如何判断用户层级？
A: 使用 `detect_user_tier` 工具，通过关键词匹配自动判断。高端关键词包括：史诗、独家、定制、奢华、秘境等。

### Q3: 风险评分如何计算？
A: 基础分100，根据天气依赖、预约要求、体力要求等因素扣分。最终分数决定风险等级和责任分级。

### Q4: 什么时候需要生成Fallback方案？
A: 当风险评分<70分时，必须调用 `generate_fallback_plan` 生成Fallback方案。

### Q5: 如何保证输出的格式一致性？
A: 在 Prompt 中明确要求输出格式，并在示例中展示标准格式。如果模型仍不遵循，考虑使用后处理或JSON输出再转换。
